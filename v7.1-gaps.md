# Lisan v7.0 → v7.1 缺失功能清单

> 对比来源：bloodscale v7.1（OpenClaw workspace-writer）
> 生成时间：2026-03-18

---

## 1. 真相文件体系（Truth Files）

### 现状
Lisan 没有任何连续性管控机制。章与章之间的世界状态、伏笔、角色信息边界完全依赖 LLM 自行记忆，容易出错。

### 缺失内容
在项目根目录下新增 `truth/` 目录，包含三个文件：

- **`truth/current_state.md`** — 世界状态快照
  - 主角位置、身体状态、持有物品
  - 关键NPC当前状态
  - 势力格局
  - 时间线

- **`truth/pending_hooks.md`** — 伏笔追踪表
  - 活跃伏笔（编号、描述、埋设章节、预计回收）
  - 已回收伏笔
  - 滞留伏笔（>10章未回收，自动标记）

- **`truth/character_matrix.md`** — 角色交互矩阵
  - 每个角色知道什么 / 不知道什么（信息边界）
  - 角色间关系状态
  - 最后交互章节号

### 影响范围
- `WritePipeline`：起草前需读取真相文件做自检，Data Agent 步骤需更新真相文件
- `ContextAgent`：执行包中需包含真相文件摘要
- `init` 命令：需创建 truth/ 目录和模板文件
- `BookConfig` 或项目配置：需声明 truth 文件路径

## 2. 写前自检 / 写后结算（PRE_WRITE_CHECK / POST_SETTLEMENT）

### 现状
Lisan 的 Draft Agent 直接根据执行包起草正文，没有任何自检环节。写完也没有结算，Data Agent 只做实体提取和摘要。

### 缺失内容

**写前自检（PRE_WRITE_CHECK）**：起草 Agent 动笔前，先输出一个结构化检查块：
1. 上下文确认：本章场景列表、情绪任务、爽点类型
2. 世界状态核对：主角位置/状态/物品（对照 current_state.md）
3. 信息边界检查：出场角色各自知道/不知道什么（对照 character_matrix.md）
4. 伏笔检查：本章需回收/新埋的伏笔（对照 pending_hooks.md）
5. 风险扫描：可能出错的点

**写后结算（POST_SETTLEMENT）**：正文写完后，追加结算块：
1. 角色交互变动：首次见面、新信息获知、关系变化
2. 伏笔变动：新埋/回收了什么
3. 世界状态变动：位置、物品、身体状态、势力变化
4. 升级事件：能力提升/新技能/资源获取

### 影响范围
- `WritePipeline` Step 2（Draft Agent）：prompt 需改为三阶段（自检 → 正文 → 结算）
- `WritePipeline` Step 5（Data Agent）：需读取 POST_SETTLEMENT 块来更新真相文件
- `AgentExecutor` 或 Draft Agent 的 systemPrompt：需注入自检/结算指令

## 3. 确定性后验证器（Post-Write Checker）

### 现状
Lisan 起草完直接进入 5 Pass 重写，没有任何非LLM的规则检查。AI 常见的文风问题（套话、重复句式、元叙事等）全靠 Pass 5 Anti-AI 用 LLM 去抓，成本高且不稳定。

### 缺失内容
在重写管线之前插入一步零LLM成本的确定性规则检查，输出违规清单注入后续 Pass 的 prompt。

检查项（11条规则）：
1. "不是……而是……"句式频率
2. 破折号使用频率
3. 转折标记词频率（然而/但是/不过/却）
4. 元叙事检测（"这个世界"/"在这片大陆"等上帝视角用语）
5. 分析术语检测（"本质上"/"从某种意义上"等论文腔）
6. 说教词检测（"应该"/"必须"/"不得不"等）
7. 集体震惊检测（"所有人都"/"众人纷纷"等）
8. 情绪标签检测（"他感到愤怒"/"她心中一喜"等直接命名情绪）
9. 段落等长检测（连续段落字数过于接近）
10. 连续相同开头检测（连续段落以相同词开头）
11. 连续"了"字检测（句尾连续出现"了"）

### 实现方式
- 独立脚本或模块，输入章节文件路径，输出 JSON 格式的 errors + warnings
- 在 WritePipeline 的 Draft Agent 之后、Pass 链之前插入
- 违规清单注入每个 Pass 的 prompt 作为额外修复指令
- errors = 0 且 warnings = 0 时跳过注入

### 影响范围
- `WritePipeline`：在 Step 2 和 Step 3 之间新增 Step 2.5
- `RewritePipeline`：在读取正文和 Pass 链之间新增同样的检查
- 新增模块/脚本：`post-write-checker`（纯字符串/正则操作，不依赖 LLM）
- `PassRunner` 或 Pass 接口：需支持注入额外 context（违规清单）

---

## 优先级建议

| 功能 | 优先级 | 理由 |
|------|--------|------|
| 确定性后验证器 | 高 | 零成本，立即可用，效果明显 |
| 真相文件体系 | 高 | 连续性是长篇小说的命脉 |
| 写前自检/写后结算 | 中 | 依赖真相文件，需先实现真相文件 |
