# 血色天平写作系统 — 工作流全景文档

> 本文档完整描述莎士比亚（writer-2 agent）从大纲到完稿的全部工作流，包括设计思想、每一步操作、使用的技术手段。供软件实现对照校正。

---

## 一、设计哲学（为什么这样做）

### 1.1 核心问题：大纲和正文之间缺了一层

大纲是粗粒度的（"卡列尔在阴沟遭遇清道夫势力"），正文是细粒度的（3000-4000字的沉浸式叙事）。直接从大纲写正文，AI 会：
- 把大纲当流水账翻译，没有情绪节奏
- 不知道哪里该压制、哪里该释放
- 每章的情绪曲线是平的

**解决方案：三层架构**
```
大纲（已有） → 场景树（中间层） → 正文（执行层）
```

场景树是关键创新——它把大纲拆成带情绪标记的小场景，每个场景自带：
- 情绪任务（读者在这里应该感受什么）
- 压制/释放标记（蓄压、蓄压加深、小释放、大释放）
- 事件骨架（具体发生什么）
- 摄像机焦点（从谁的感官出发）

这样起草 Agent 拿到的不是"写第12章"，而是"这一章包含场景 S003-S005，情绪曲线是蓄压→蓄压加深→小释放，爽点类型是怒火宣泄"。

### 1.2 体验优先，不是正确性优先

优先级排序：**读者情绪体验 > 设定正确性 > 文笔**。

四条铁律：
1. **情绪不能断**：每段文字都必须服务于读者的情绪体验
2. **摄像机在体内**：从角色感官出发，禁止全知旁白
3. **无地球价值观泄露**：角色不能暗示人权、平等、民主等概念
4. **暴力即美学**：极致残酷热血，"好狠但好爽"

### 1.3 主会话纯调度，执行全部下放

Claude（主会话）只做流程编排和调度，不读正文、不写正文。所有实际执行（起草、重写、数据处理）都通过 `sessions_spawn` 派发给 GPT 子代理（helper agent）。

原因：
- 主会话保持轻量，不被长文本污染上下文
- 子代理可以独立超时、独立失败，不影响主流程
- 不同任务可以用不同的 prompt 和超时配置

### 1.4 真相文件系统：解决跨章连续性

AI 写长篇小说最大的问题是"失忆"——第20章忘了第5章发生过什么。

解决方案：三个真相文件（`{ROOT}/truth/`），每章由 Data Agent 自动更新：
- `current_state.md`：当前世界状态（主角位置、身体状态、持有物品、势力格局）
- `pending_hooks.md`：活跃伏笔清单（埋了什么、回收了什么、哪些滞留太久）
- `character_matrix.md`：角色交互矩阵（谁知道什么、谁见过谁、信息边界）

起草前的"写前自检"会读这三个文件，确保不出现角色知道不该知道的信息、物品凭空出现等问题。

## 二、技术基础设施

### 2.1 运行时环境

| 组件 | 说明 |
|------|------|
| 主调度 | OpenClaw writer-2 agent（Claude），纯调度不执行 |
| 执行代理 | helper agent（GPT-5.4），通过 `sessions_spawn(agentId:"helper", mode:"run", runtime:"subagent")` 启动 |
| 项目管理 | webnovel.py 脚本（Python），处理 preflight、workflow 状态、实体提取、摘要、RAG 索引 |
| 上下文数据库 | OpenViking（ov CLI），语义检索 + 分层读取（L0/L1/L2） |
| 向量嵌入 | 阿里云 DashScope text-embedding-v3 |
| 版本控制 | Git，每章完成后自动 commit |
| 确定性检查 | post_write_check.py，零 LLM 成本的规则检查器 |

### 2.2 OpenViking（ov）— 上下文管理的核心

**问题**：小说项目文件很多（大纲30个文件、设定集、角色库、已完成章节），不可能全部塞进 prompt。

**解决方案**：OpenViking 提供三层渐进式读取：
- **L0 abstract**：一句话摘要（~200字）— 用于快速判断是否需要这个文件
- **L1 overview**：结构化概览（~1-2KB）— 用于了解文件大意
- **L2 read**：完整原文 — 仅在确认需要时使用

**语义检索**：`ov find "关键词" -u viking://resources/bloodscale -n 5`，返回最相关的文件片段。

**规则**：先 L0/L1 判断是否需要，确认需要才读 L2。禁止无脑读 L2 全文。>5KB 的文件必须通过 ov 获取，不能直接从磁盘读。

**数据结构**：
```
viking://resources/bloodscale/
├── 章纲/       ← 大纲按阶段拆分
├── 场景树/     ← scenes.md + chapter-plan.md
├── 设定集/     ← 全部设定+角色库
├── references/ ← 写作参考文件
└── 正文/       ← 已完成章节
```

每章完成后、场景树生成后、章节规划生成后，都会同步到 ov。

### 2.3 webnovel.py — 项目管理脚本

统一入口：`python -X utf8 "D:/tmp/webnovel-writer/webnovel-writer/scripts/webnovel.py" --project-root "{ROOT}" <cmd>`

主要命令：
- `preflight --format json`：检查项目状态
- `workflow start-task --command <cmd> --chapter <N>`：开始任务
- `workflow complete-task --artifacts <json>`：完成任务
- `context --chapter N`：获取章节上下文
- `rag index-chapter`：向量索引（需要 DashScope 环境变量）
- `entity lookup-all --mention <text>`：实体查询

### 2.4 post_write_check.py — 确定性后验证器

零 LLM 成本的规则检查器，在重写管线之前运行。检查项包括：
- 惊讶标记词密度（"仿佛"、"忽然"、"竟然"等）
- 元叙事模式（"接下来就是"、"故事发展到了"等）
- 报告体术语（"核心动机"、"信息边界"等 AI 味词汇）
- 说教词汇（"显然"、"毋庸置疑"等）

输出 JSON 格式的违规清单（errors + warnings），注入到重写 Pass 的 prompt 中作为额外修复指令。

### 2.5 参考文件体系

```
skills/bloodscale/references/
├── core/
│   ├── character-quick-ref.md    ← 角色速查卡（行为模式表）
│   ├── core-constraints.md       ← 核心约束（铁律）
│   └── experience-design.md      ← 体验设计总纲（四种爽点）
├── craft/
│   ├── polish-guide.md           ← 打磨指南
│   └── typesetting.md            ← 排版规范
└── style/
    ├── combat-writing.md         ← 战斗写作指南
    ├── protagonist-voice.md      ← 主角声音
    └── worldview-and-characters.md ← 世界观与角色
```

不同 Agent 按需读取不同的参考文件，不是所有 Agent 都读所有文件。

## 三、完整工作流（从大纲到完稿）

### 阶段 A：场景分解（/webnovel-decompose）

**输入**：一个弧线（阶段）的章纲文件
**输出**：该弧线的场景树（scenes.md）
**执行者**：helper 子代理（scene-decomposer agent）

#### 步骤：

1. **Preflight**：启动 OV 服务，确认章纲文件存在，创建场景树目录
2. **Spawn 场景分解器**：
   - 读取弧线章纲（磁盘直读）
   - 读取总纲定位（磁盘直读）
   - 读取体验设计总纲（磁盘直读）
   - 非首弧时读取前弧衔接摘要
   - 输出完整场景树到 `{ROOT}/场景树/arc-{NN}-{弧线名}/scenes.md`
3. **OV Sync**：将 scenes.md 同步到 ov
4. **人工审查**：输出摘要，等待用户确认/修改

**关键设计思想**：
- 每个场景必须有情绪任务，说不出"读者在这里感受什么"的场景不该存在
- 蓄压场景不是废话场景，每个蓄压都要比上一个更紧
- 按弧线批量执行，审查完一个弧线后该弧线内章节可连续自动跑

---

### 阶段 B：章节规划（/webnovel-plan）

**输入**：审查通过的场景树
**输出**：章节规划（chapter-plan.md）
**执行者**：helper 子代理

#### 步骤：

1. **Spawn 章节规划师**：
   - 读取场景树 scenes.md（磁盘直读）
   - 读取人工批注 notes.md（如有）
   - 读取体验设计总纲
   - 将场景分配到章节，每章 2-4 个小场景或 1-2 个中场景
   - 输出到 `{ROOT}/场景树/arc-{NN}-{弧线名}/chapter-plan.md`
2. **OV Sync**：将 chapter-plan.md 同步到 ov
3. **通知用户**：输出摘要，等待确认

**章节规划的输出格式**（每章包含）：
- 包含场景列表
- 章级情绪任务（一句话）
- 情绪曲线（起点→压制→释放）
- 爽点类型
- 预估字数
- 章末钩子

**关键规则**：
- 每章必须有独立的情绪弧线，全章平线 = 失败
- 章末必须有钩子让读者想翻下一章
- 过渡场景跟蓄压场景合并，不单独成章
- 大释放场景可以独占一章

---

### 阶段 C：正文写作（/webnovel-write）— 单章完整流程

确认后进入逐章执行。每章经历以下 7 个步骤：

#### Step 0: Preflight

```
1. 启动 OV 服务（PowerShell 脚本）
2. 运行 preflight 检查项目状态
3. 注册 workflow 任务
4. 确认该章的章节规划存在
```

#### Step 1: Context Agent（生成创作执行包）

**Spawn helper**（超时 300s）

执行包是起草 Agent 的唯一输入，包含：
1. **场景蓝图**（最高优先级，执行包第一段）：
   - 章级情绪任务、情绪曲线、爽点类型、章末钩子
   - 每个场景的完整定义（情绪任务、事件骨架、摄像机焦点、压制/释放标记）
2. **上章衔接**：上一章最后 500 字
3. **设定参考**：通过 ov find 按需查询的设定摘要
4. **出场角色**：角色卡摘要，新角色则生成角色卡

输出到：`{ROOT}/.webnovel/tmp/context_pack_ch{N:04d}.md`

**关键思想**：起草 Agent 不需要知道大纲全貌，它只需要知道"这一章要完成哪些场景、每个场景的情绪任务是什么"。执行包是一个自包含的创作指令。

#### Step 2: 正文起草

**Spawn helper**（超时 300s）

起草 Agent 的读取顺序（严格按序）：
1. 执行包（场景蓝图在第一段）
2. 角色速查卡（行为模式表）
3. 核心约束（铁律）
4. 三个真相文件（世界状态、伏笔、角色矩阵）
5. 主角声音（ov overview）
6. 世界观（ov overview）
7. 战斗章额外读战斗写作指南

**三阶段写作**：

**Phase 1 — 写前自检（PRE_WRITE_CHECK）**：
动笔前在文件开头写 HTML 注释块，包含：
- 上下文确认（场景列表、情绪任务、爽点类型）
- 世界状态核对（对照 current_state.md）
- 信息边界检查（对照 character_matrix.md）
- 伏笔检查（对照 pending_hooks.md）
- 风险扫描

**Phase 2 — 正文写作**：
- 按场景序列写，每个场景完成它的情绪任务
- 摄像机锁定在主角体内
- 感官优先级：触觉/痛觉 > 听觉 > 嗅觉/味觉 > 视觉
- 情绪通过身体反应和行为传递，禁止情绪形容词
- 对话极简，主角全章不超过 5 句完整的话
- 章末落在执行包指定的钩子上
- 目标字数：3000-4000 字

**Phase 3 — 写后结算（POST_SETTLEMENT）**：
正文末尾追加 HTML 注释块，包含：
- 角色交互变动
- 伏笔变动
- 世界状态变动
- 升级事件

输出：`{ROOT}/正文/第1卷/第{N:04d}章.md`（单次 write 写完整章）

#### Step 2.5: 确定性后验证（零 LLM 成本）

运行 `post_write_check.py`，输出违规清单（JSON）。
- errors + warnings > 0 → 注入到重写 Pass 的 prompt 中
- errors = 0 且 warnings = 0 → 正常进入重写管线

**意义**：用确定性规则（正则匹配、词频统计）抓 AI 味词汇和结构问题，不消耗 LLM token。

#### Step 3: 5 Pass 重写管线

**严格按顺序执行，每个 Pass 一个 spawn，每个 Pass 直接 edit in-place 修改正文文件。**

| Pass | Agent | 职责 | 超时 |
|------|-------|------|------|
| 1 | experience-implant | 体验植入：把"在外面看"改成"在里面活" | 240s |
| 2 | thrill-boost | 爽点强化：压制压得更狠，释放炸得更响 | 240s |
| 3 | rhythm-tension | 节奏张力：该快如子弹，该慢如刀割 | 180s |
| 4 | dialogue-combat | 对话博弈：每句话都是一次出招 | 180s |
| 5 | anti-ai | Anti-AI 终检：消灭一眼 AI 的痕迹 | 180s |

**每个 Pass 的输入**：
- 正文文件（磁盘直读）
- 角色速查卡
- 体验设计总纲
- Pass 1/2 额外读执行包（了解场景情绪任务）
- Pass 3 额外读战斗写作指南

**写入策略**：禁止 write 覆盖整个文件，必须用 edit 逐处修改。修改完成后在文件末尾追加修改报告（HTML 注释）。

**关键思想**：每个 Pass 只关注一个维度，避免"什么都改一点但什么都改不好"。顺序有讲究——先植入体验（最重要），再强化爽点，再调节奏，再磨对话，最后去 AI 味。

#### Step 4: 终审 + Progression

**Spawn helper**（超时 300s）

四部分审查：
- **Part A — 场景完成度审查**（最高优先级）：每个场景是否展开、情绪任务是否完成、章末钩子是否落位
- **Part B — 体验审查**：摄像机位置、感官传递、角色行为一致性
- **Part C — 设定与排版审查**：设定矛盾、排版规范、地球价值观泄露
- **Part D — Progression 判断**：是否有能力提升/新技能/资源获取，有则更新主角卡+升级日志

发现问题直接 edit 修复，不是只报告。

#### Step 5: Data Agent（四合一）

**Spawn helper**（超时 300s）

四个 Part 按序执行：
- **Part A**：实体提取（通过 webnovel.py）
- **Part B**：章节摘要生成
- **Part C**：向量嵌入（RAG 索引，需要 DashScope 环境变量）
- **Part D**：真相文件更新（核心）
  - 读取 POST_SETTLEMENT 注释块
  - 更新 `current_state.md`（世界状态）
  - 更新 `pending_hooks.md`（伏笔追踪）
  - 更新 `character_matrix.md`（角色交互矩阵）

**关键思想**：真相文件是下一章写前自检的依据，必须准确。宁可多记不可漏记。

#### Step 6: OV Sync

将完成的正文同步到 OpenViking：
```
ov add-resource "{正文文件}" --parent viking://resources/bloodscale/正文 --reason "第N章正文" --wait --timeout 120
```

#### Step 7: Git + 完成

```
git add . && git commit -m "第N章: {标题}"
workflow complete-task
```

---

### 阶段 D：批量写作（/webnovel-batch N）

循环调用阶段 C 的完整流程，连续写 N 章。主会话纯调度，不读正文。

通常批量 ~10 章后强制暂停，等待人工审阅和休息。

---

### 阶段 E：重写（/webnovel-rewrite）

对已有初稿只跑重写管线，跳过起草。流程：
Step 1 → 5 Pass 重写 → 终审 → Data Agent → OV Sync → Git

## 五、关键技术决策清单

| 决策 | 原因 |
|------|------|
| 三层架构（大纲→场景树→正文） | 大纲太粗，直接写正文情绪曲线是平的 |
| 场景自带情绪任务 | 让 AI 知道"读者在这里应该感受什么"，而不是"这里发生了什么" |
| 执行包是自包含的 | 起草 Agent 不需要了解全书，只需要知道这一章的任务 |
| 主会话纯调度 | 避免长文本污染上下文，子代理独立超时/失败 |
| 5 Pass 串行重写 | 每个 Pass 只关注一个维度，避免"什么都改一点" |
| edit in-place 而非 write 覆盖 | 重写是精修，不是重写整篇；覆盖会丢失前面 Pass 的修改 |
| 真相文件系统 | 解决 AI 跨章失忆问题，每章自动更新 |
| 写前自检 + 写后结算 | 起草 Agent 自己检查连续性，不依赖外部审查 |
| 确定性后验证器 | 零 LLM 成本抓 AI 味词汇，在重写前就发现问题 |
| OV 分层读取 | 避免把所有文件塞进 prompt，按需获取上下文 |
| 人工审查按弧线批量 | 审完一个弧线后该弧线内章节可连续自动跑，减少人工干预频率 |
| 每章 Git commit | 版本控制，可回滚 |
| POST_SETTLEMENT 注释块 | 起草 Agent 在写完时就记录变动，Data Agent 据此更新真相文件，形成闭环 |

## 六、子代理（spawn）规范

所有执行通过 `sessions_spawn(agentId:"helper", mode:"run", runtime:"subagent")` 启动。

每个 spawn 必须：
1. 设置 `runTimeoutSeconds`（240-300s 不等）
2. task prompt 末尾加防停顿指令：`⚠️ 完成所有步骤后才停止。不要在中间步骤等待确认。`
3. task prompt 中包含 ov CLI 使用规范（怎么搜索、怎么分层读取）
4. 加粗关键红线（2-3 条）
