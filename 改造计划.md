# Lisan 主链路改造计划（Desktop + Engine + Tauri Sidecar）

> 评估范围与本计划一致：仅覆盖当前真实主链路，不展开 deprecated 的 `packages/cli` / `packages/core`。

## 1. 目标与验收口径

### 1.1 总目标
1. 打通“拆解场景 -> 章节执行 -> 正文落地”闭环。
2. 让执行控制（至少 `abort` / `skip`）在运行中真实生效。
3. 让项目首页、章节页、执行页的数据状态一致且可解释。

### 1.2 商业可用门槛（必须先过）
1. 章节工作流完成后自动回写章节正文。
2. `abort/skip` 能中断正在执行的当前步骤，而不是只改状态位。
3. 场景拆解结果与章节强绑定，章节上下文可稳定拿到场景。

- [x] `AgentExecutor.execute()` 增加 `signal` 参数并透传到 LLM Provider。
- [x] `abort(executionId)` 调用对应步骤 `controller.abort()`，并终止后续步骤。

## 6. 风险与回滚

### 6.1 主要风险
1. `abort/signal` 透传会影响现有 provider 重试逻辑（需重点回归）。
2. 场景章节绑定修复涉及历史数据迁移，需避免误绑。
3. 首页聚合从文件转 DB 后，老项目兼容性可能受影响。

### 6.2 回滚策略
1. 以任务粒度小步提交，每个任务独立可回退。
2. 对外行为变更（skip/abort、状态枚举）先加 feature flag（可选）。
3. 数据迁移脚本提供 dry-run 与备份开关。

---

## 7. 最终 DoD（Definition of Done）

1. P0 三项门槛全部通过（正文回写、控制生效、章节绑定）。

3. 首页、章节页、执行页三处状态一致，且可从 DB 追溯。
4. RAG 入口不再“看起来可用但实际不可用”。
5. Data Agent / checker / truth 至少完成一条端到端可观测链路。
